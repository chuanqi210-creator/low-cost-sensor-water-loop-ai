# 可量化终止条件版 Goal 扩展

## 你的问题被重新表述为

当前模型迭代方向是对的，但缺少“什么时候算阶段完成”的计算规则。如果没有终止条件，系统容易陷入两种低效：一是不断新增 agent、字段和治理文件，却不知道是否真正提升模型成熟度；二是每次自我打断都重算上下文，增加摩擦，却没有形成可验收的小闭环。

因此，后续 goal 需要从“持续优化模型”升级为“按照可计算成熟度门槛推进模型”。每轮迭代不只问做了什么，还要问：核心能力分数提升了多少，硬门槛是否通过，是否达到阶段收束条件，下一步继续做的边际价值是否仍然高于停下整理或等待真实数据。

## 外部框架借鉴

本项目不直接照搬外部成熟度体系，但借鉴其思想：

- TRL/TRA：技术不能只停留在概念描述，必须逐步进入实验、相关环境验证、真实环境验证。
- MRL：工程化不只看算法能否运行，还要看制造、部署、运维、成本、质量和供应链约束是否可进入真实系统。
- V&V：verification 问“系统是否按设计做对了”，validation 问“设计是否满足真实需求”。
- 专利交底：技术方案必须能拆成技术问题、技术手段、实施例、技术效果、区别于现有技术的特征和失败边界。

这些框架共同提醒我们：迭代必须有证据等级、成熟度等级和阶段门，而不是靠主观感觉说“更好了”。

## 总体终止思想

本 goal 不追求无限迭代。它追求的是：在每个阶段，把系统推进到一个可验证、可解释、可交接的成熟度门槛；一旦达到门槛，就停止当前阶段的细节扩张，进入下一阶段或等待真实数据。

终止不是停止项目，而是停止某一轮、某一模块、某一阶段的继续堆叠。终止条件本质上是防止模型碎片化、防止重复治理、防止把 synthetic 结果误当 field 结论。

## 七层骨架的量化评分

每轮迭代都必须映射到七层系统骨架，并给出 0 到 5 分评分：

1. 现场对象层：污染物、水质、处理单元、管网/反应器、循环结构、催化剂/药剂/膜/生物单元是否被结构化表达。
2. 观测层：低成本传感、稀疏布点、node-modality 矩阵、缺测、延迟、噪声和路径覆盖是否被表达。
3. 状态估计层：软传感器是否能估计隐藏状态，并输出不确定性、外推风险和失败边界。
4. 机理证据层：知识库、文献 source_basis、灰箱机制和证据等级是否约束判断。
5. 诊断决策层：多智能体是否形成可解释诊断、候选动作、仲裁和人工复核边界。
6. 闭环执行层：回流、延长停留、暂存、投药、切换单元、保护阻断和 release gate 是否可执行。
7. 验证治理层：replay、holdout、field package、operator review、claim gate 和日志追踪是否完整。

评分含义：

- 0 分：该层缺失。
- 1 分：只有文字概念。
- 2 分：有 schema 或接口，但没有测试。
- 3 分：有 synthetic 验证和测试。
- 4 分：有可接真实数据的 package/gate/replay 接口。
- 5 分：有真实数据、holdout、人工复核或现场回放验证。

阶段性系统骨架终止条件：

- 七层均不低于 3 分。
- 观测层、状态估计层、闭环执行层、验证治理层不低于 4 分。
- 任何 2 分以下的层不能阻断主链理解，否则必须优先补齐。

## 七类核心能力评分

每轮迭代还要给七类能力打分，每类 0 到 5 分：

1. 可观测性：黑箱是否更接近灰箱，关键隐藏状态是否更可估计。
2. 可控性：低频、低成本、延迟观测下是否仍能选择可执行动作。
3. 可解释性：诊断和动作是否能追溯到状态、机制和证据路径。
4. 可验证性：synthetic、literature、template、field、operator-reviewed、release-gated 是否分清。
5. 可工程化：数据采集、校准、执行器约束、成本和人工复核是否可落地。
6. 可演化性：是否减少碎片、合并冗余、稳定接口，而不是继续堆 agent。
7. 可保护性：是否形成具体技术特征、实施例、技术效果和现有技术区别。

核心能力总分计算：

`core_score = 0.18*可观测性 + 0.16*可控性 + 0.14*可解释性 + 0.18*可验证性 + 0.14*可工程化 + 0.10*可演化性 + 0.10*可保护性`

单轮有效迭代终止条件：

- `core_score` 比上一轮提升至少 0.10，或解决一个 P1/P2 硬阻断。
- 没有新增 synthetic/field 边界混淆。
- 至少新增或修复一个可测试接口、指标、gate 或证据边界。

如果单轮 `core_score` 提升小于 0.05，且没有解决硬阻断，该轮应停止扩展，进入复盘或低优先级 backlog。

## 模块成熟度门槛

每个核心模块必须达到以下可计算门槛，才算当前阶段完成：

- 输入契约完整率 >= 0.95。
- 输出契约完整率 >= 0.95。
- 状态变量覆盖率 >= 0.90。
- 下游消费者回接率 >= 0.80。
- 测试覆盖状态：至少有 targeted tests；关键控制链必须进入 replay 或 holdout gate。
- 证据边界完整率 = 1.00，即必须明确 synthetic、literature、template、field 不能互相替代。
- 失败边界完整率 >= 0.90。
- no-write 边界必须明确：什么时候不能写 actuator，什么时候不能写 release gate。

模块终止条件：

当模块达到上述门槛，且下一步改动只会增加细节而不能增强主链时，停止该模块扩张，转向上游根因或下游回接。

## 专利级成熟度门槛

本项目的“专利级”不是保证授权，而是用专利交底标准约束技术方案清晰度。专利级阶段门按 0 到 5 分评价：

1. 技术问题清晰度：是否具体到低成本受限观测、循环争取时间、灰箱状态估计、闭环控制等真实工程问题。
2. 技术手段具体度：是否有传感布点、特征矩阵、软传感器、路径合同、诊断仲裁、执行动作和验证门。
3. 实施例完整度：是否至少覆盖实验室循环反应器、低成本现场包、多设施/管网协同三个实施例。
4. 技术效果可测性：是否有 baseline、指标、阈值、对照策略和失败边界。
5. 现有技术区别度：是否说明区别于单纯软传感、单纯 MARL、单纯稀疏布点、单纯循环优化和单纯 AI 催化剂发现。
6. 证据等级诚实度：是否明确哪些只是 synthetic，哪些需要 formal search、field replay 和人工复核。

专利交底草案 readiness 条件：

- 六项均 >= 4 分。
- 已形成 claim skeleton、technical feature ledger、embodiment matrix、technical effect matrix、prior-art distinction matrix。
- 已生成 formal search work packages 和 result intake schema。
- 仍必须标注：formal search 未完成前不能给出新颖性/创造性法律结论。

## 论文成熟度门槛

论文 readiness 分为两类：

方法论文 readiness：

- 有清晰问题定义、系统架构、算法链、数据流和评价指标。
- 有 synthetic benchmark、baseline comparison、ablation 或 counterfactual replay。
- 有失败边界和 field validation 计划。
- 允许投稿为方法/框架/仿真论文，但不能声称真实现场效果。

实证论文 readiness：

- 有真实 field package。
- 有 field holdout、replay、operator review 或 release validation。
- 有真实污染物/水质/工况标签。
- 有至少一个现场或半现场系统验证。

若没有真实数据，系统只能达到方法论文或专利交底草案成熟度，不能达到实证论文成熟度。

## 工程成熟度门槛

工程化 readiness 不能只看模型能不能跑，还要看能不能接现场：

- Field package schema 完整。
- 必需表、字段、时间戳、batch_id、node_id、sensor_id、operation log、offline lab results 可对齐。
- 低成本传感器布点能映射到 hydraulic path stage。
- 软传感输出具备不确定性和 OOD 风险。
- 控制动作具备执行器约束、药剂库存、池容、泵阀动作次数、人工复核时间。
- replay 通过前不能写 actuator。
- release gate 通过前不能放行。

工程阶段终止条件：

- 若真实数据缺失，工程阶段终止在 package/gate/template/handoff 完成，不继续伪造 field 结论。
- 若真实数据已提供，则进入 import、replay、holdout、operator review、protective candidate、final execution review 的顺序门。

## 自我打断的节流规则

自我打断不是模仿用户频繁打断，而是低频治理阀门。只有以下情况才立即打断：

- 工作偏离七层骨架，开始服务展示、整理或局部美化，而不是核心模型。
- 把 synthetic/template/literature 结论写成 field 结论。
- 绕过 replay、holdout、operator review、release gate 或 actuator no-write 边界。
- 新增复杂度没有提升接口清晰度、证据链、可控性或验证能力。
- 当前小闭环已完成，继续做同一模块的边际收益低于切换到下一个高优先级问题。

普通新想法不立即打断，先进入阶段边界 backlog。每完成一个可验证小闭环，才做一次深度复盘和优先级重排。

## 阶段终止判定

每轮结束时必须输出：

- 本轮属于哪一层。
- 本轮增强哪类能力。
- 本轮修改了哪些输入、输出、状态变量、指标、证据边界或 gate。
- 本轮 `core_score` 是否提升。
- 是否解决 P1/P2 阻断。
- 是否还有 field blocker。
- 是否允许进入下一阶段。
- 若不继续，原因是达到阶段门、等待真实数据、边际收益过低，还是存在硬阻断。

一个阶段可以终止，当满足：

- 当前最高优先级问题已经有可测试接口或可执行补包方案。
- targeted tests 通过。
- 完整回归通过或明确说明未运行原因。
- 产物、状态日志和 manifest 已同步。
- synthetic/field/patent/legal 边界没有混淆。
- 下一步最高边际价值动作已经明确。

## 当前项目适用的主终止目标

本项目当前主终止目标不是“做完所有 agent”，而是达到以下阶段：

1. 系统骨架成熟：七层主链可追踪，核心模块接口稳定。
2. 模型核心成熟：稀疏感知、软传感、灰箱机理、多智能体诊断、闭环控制、field gate 形成可运行链路。
3. 工程接口成熟：真实 field package 能被导入、检查、回放和阻断。
4. 证据治理成熟：所有结论能区分 synthetic、template、literature、field、operator-reviewed、release-gated。
5. 专利交底成熟：能形成技术问题、技术手段、实施例、技术效果、区别现有技术和失败边界的完整草案。

当 1 到 5 都达到门槛，且没有真实数据时，应停止继续堆代码，转向真实数据采集、formal search、人工审查和 field validation。

## 参考来源

- NASA/GAO 等技术成熟度评估思想用于提醒“技术必须从概念、实验验证走向相关环境和真实环境验证”。
- MRL/工程制造成熟度思想用于提醒“工程化不能只看算法运行，还要看部署、质量、成本和运维”。
- Systems engineering V&V 思想用于区分“系统是否按设计做对”和“设计是否满足真实需求”。
- WIPO/专利交底实践用于反推技术方案必须具备技术问题、技术手段、实施例、技术效果和现有技术区别。
